Road Sense Project Guide

A simple overview of the hardware, software, and logic powering the Road Sense platform.

🛠️ Hardware Used

1. ESP32-C3 Microcontroller
What it is: The "brain" of the project.
What it does: It's a tiny, powerful computer with built-in Wi-Fi. It runs the C++ code, communicates with the sensor to read bump data, and connects to the internet to send that data to your web dashboard.
2. MPU6050 (GY-521) Sensor
What it is: An Accelerometer and Gyroscope sensor.
What it does: It measures movement and g-force in 3 directions (Up/Down, Left/Right, Forward/Backward). This is how the system feels "bumps" and "potholes" on the road.
3. Custom PCB & Buck Converter
What it is: The physical hardware foundation.
What it does: Instead of fragile jumper wires, a Custom Printed Circuit Board (PCB) securely connects the ESP32 brain to the MPU6050 sensor. The board includes a 12V/24V to 5V Buck Converter, allowing it to be wired safely and directly into the vehicle's fuse box for permanent power.

💻 Software Used

1. Arduino IDE
What it is: A programming application for your computer.
What it does: We used this to write the code, compile it, and upload the instructions into the ESP32 chip. It also gives us the "Serial Monitor" to see live diagnostic messages.
2. Road Sense Web Dashboard
What it is: Your custom website interface and database.
What it does: It listens for incoming bump data from the ESP32, saves it, and displays it in real-time so users can monitor road infrastructure issues.
3. Render (Cloud Hosting)
What it is: A cloud hosting platform.
What it does: It hosts your web dashboard on the internet 24/7. This allows your ESP32 to send data securely to your dashboard no matter where it is, as long as it has an internet connection.

🧠 Code Explanation (How it Works)

The code running on the ESP32 is responsible for continuously reading the road surface and deciding if an anomaly (like a pothole) occurred. Here is a simplified breakdown of the core logic.

1. Reading the Raw Acceleration

The ESP32 asks the MPU6050 sensor for the current acceleration on all 3 spatial axes. The sensor measures this in "g-force" (where 1.0g is standard Earth gravity).

  • ax (X-axis: Left/Right)
  • ay (Y-axis: Forward/Backward)
  • az (Z-axis: Up/Down)

2. Calculating the "Total Magnitude"

When the vehicle hits a bump, the sensor shakes in multiple directions at once. To figure out the overall force of the bump, the code uses a 3D Pythagorean theorem formula to combine all three axes into a single magnitude number.

float magnitude = sqrt(ax * ax + ay * ay + az * az);

Sitting perfectly still, this number is ~1.0g. When a bump occurs, this number spikes sharply (e.g., to 2.5g or 4.0g).

3. Categorizing the Severity

Once the code has the magnitude, it compares it against three predefined thresholds to categorize the severity of the bump:

if (magnitude > 3.0) {
    severity = "high"; // Deep Pothole
} else if (magnitude > 2.0) {
    severity = "medium"; // Surface Crack
} else if (magnitude > 1.5) {
    severity = "low"; // Uneven Surface
}
  • HIGH (> 3.0g): Represents a violent jolt, like dropping into a deep pothole or hitting a large object.
  • MEDIUM (> 2.0g): A noticeable, jarring bump, such as a major surface crack or speedbump hit too fast.
  • LOW (> 1.5g): A mild thud or minor unevenness in the road.

If the magnitude never reaches 1.5g, the system ignores it as normal road vibration. If a bump is detected, the ESP32 instantly packages this data and sends it over Wi-Fi via an HTTP POST request to your cloud-hosted dashboard.

🚀 Future Optimizations (Real-World Implementation)

The Road Sense ESP32 prototype relies on inertial sensors for extreme reliability in all lighting and weather conditions. The following optimizations represent the core of the edge computing firmware planned for the production release:

1. Pothole vs. Speed Bump Recognition (The "Drop" Signature)

Right now, hitting a pothole and hitting a speed bump will both register as a high-g "bump".

Optimization Strategy: Instead of just looking at the total magnitude, the firmware analyzes the Z-axis (up and down) waveform.

  • Speed Bump: The vehicle goes UP first (positive Z spike), then down.
  • Pothole: The tire drops into the hole first (a momentary drop in Z-force, almost like brief zero-gravity), followed immediately by a massive positive Z spike when the tire slams into the lip of the hole.

By writing code that looks for this exact -Z followed by +Z signature, the ESP32 can filter out intentional infrastructure like speed bumps.

2. Suspension Auto-Calibration

A 3.0g bump in a heavy municipal bus with stiff air suspension feels very different from a 3.0g bump in a standard transit van.

Optimization Strategy: Implement a moving average "baseline" in the firmware. When the vehicle is driving on a smooth road, the ESP32 calculates the average background vibration (Exponential Moving Average). It triggers an alert only when a sudden spike is mathematically higher than that specific vehicle's baseline.

3. Advanced Window Buffering & Speed Context (Spatial Debouncing)

Currently, a 2000ms cooldown is used. If a bus drives over a rumble strip, it will spam the dashboard with alerts.

Optimization Strategy: The ESP32 utilizes a circular buffer to hold 3-5 seconds of driving data in its memory. If it detects continuous heavy vibrations, it aggregates the data and classifies the entire area as a "Degraded Road Surface" rather than 10 individual "Potholes". Additionally, integrating vehicle speed data (e.g., via GPS or OBD-II) will ensure that bumps hit at 60mph vs 10mph are scaled and categorized appropriately, reducing false positives.

❓ Common Questions & Answers

1. How does the device get internet out on the road?

Answer: The current prototype uses a mobile Wi-Fi hotspot to prove the concept. However, the final production version completely avoids expensive LTE cellular modules. Instead, it stores events locally on the fleet's dashcam and automatically uploads them when the vehicle returns to base and connects to the municipal Depot Wi-Fi.

2. What happens if the vehicle temporarily loses that internet connection?

Answer: Since the architecture relies on Depot Wi-Fi upload, continuous internet is not required. The dashcam securely stores all snapshots and GPS tags locally during the route and uploads the batch when a connection is established at the end of the shift.

3. How do you know the exact GPS location of the pothole?

Answer: The production version does not need its own expensive GPS module. Instead, the ESP32 sends a hardware trigger (via GPIO or UART) to the fleet's existing smart dashcam or telematics unit, which instantly takes a snapshot and tags it with its high-accuracy GPS coordinates.

4. Why build custom hardware? Why not just use a smartphone app?

Answer: Smartphones aren't bolted to the car. If a driver drops their phone or takes a sharp turn, an app registers it as a massive pothole. Our custom hardware is physically mounted to the vehicle chassis, ensuring we only measure real road impacts with zero false positives.

5. How is the device powered in the vehicle?

Answer: The production unit uses a built-in 12V/24V to 5V buck converter. This allows it to be safely hardwired directly into the vehicle's fuse box using a fuse tap, so it boots up automatically when the engine starts, completely hidden from the driver.

6. Is it expensive to deploy this on every city bus or garbage truck?

Answer: Not at all. By reusing the fleet's existing dashcams and GPS units, our added hardware only costs about SAR 75 per vehicle. Because we upload via Depot Wi-Fi, there are zero ongoing monthly cellular data fees. It is significantly cheaper and infinitely faster than sending out manual road-inspection crews.
